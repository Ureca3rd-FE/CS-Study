# 애자일 Agile

소프트웨어 개발 기법.

## 등장 배경

초기 소프트웨어 개발 방법은 계획 중심의 프로세스.(대표적 예시로 계획에 맞춰 차례차례 흘러가는 워터폴 방식) 하지만 트렌드가 급격하게 빨리 변화하게 되어(기술이 새로 등장하는 속도가 빠름) 비지니스 사이클(제품 수명)이 짧아졌고 SW 개발의 불확실성이 높아짐.

덕분에 옛날의 전통적 개발 방법 적용이 어려워짐. -> 일단 해보고 고쳐나가는 방식 등장.

- 규칙은 적게
- 가볍게
- 바로바로 대응할 수 있도록

경량 방법론 주의자들이 노하우를 공유하고 공통점을 추려 애자일이란 방법론이 탄생.

## 그래서 애자일이란?

`반복적 개발`과 `점진적 개선`으로 피드백을 빠르게 반영해 제품을 지속적으로 발전시키는 접근 방식

### 애자일 실천 방법

- 팀원 개개인이 자율성을 가지고 업무를 수행할 수 있어야 함(각자의 책임감 중요)
- 투명한 커뮤니케이션(모든 정보는 개방적으로 공유)
- 상명하달식 리더십보단 지원과 조율 중심의 리더십

### 하위 방법론

1. 스크럼 Scrum

시간 단위인 `스프린트`(2~4주) 기준으로 팀 전체가 정해둔 목표를 달성. 목표를 달성하기 위한 작업을 `백로그`라고 정의하여 우선순위 설정.

스프린트 중에는 매일 스탠드업 미팅을 진행해 상황 공유. 스프린트가 끝나면 해당 스프린트에서 개발한 제품의 작동 여부 검증.

#### 용어
- `스프린트`: 작업 주기(보통 1~4주)
- `백로그`: 개발할 기능 또는 작업 목록
    - 스프린트 백로그: 해당 스프린트의 작업을 우선순위에 따라 정리
    - 제품 백로그: 제품의 모든 요구사항을 우선순위에 따라 정리
- 스크럼 마스터: 스크럼 프로세스를 관리하고 스크럼 원칙을 따르도록 지원하는 역할
- 프로덕트 오너: 고객 또는 사용자와의 의사소통을 중개하고 제품의 우선순위를 결정하는 역할
- 스크럼 이벤트: 스크럼 프로세스에서 일어나는 이벤트
    - 스프린트 계획 회의: 스프린트 진행 전, 해당 스프린트에서 해야 할 작업 선정 회의
    - `데일리 스크럼 회의`: 매일 진행되는 짧은 회의로 진행 상황과 문제를 공유
    - 스프린트 리뷰 미팅: 해당 스프린트에서 개발한 제품의 작동 여부를 검증하는 회의
    - 스프린트 회고 미팅: 해당 스프린트에서 진행한 프로세스와 문제점을 검토하고 개선점을 도출하는

2. 칸반 Kanban

시각적 도구를 활용해 업무 흐름을 한눈에 파악하게 하고 병목 현상을 조기에 발견, 해결하는 데에 중점.

> 병목 현상<br>
> 전체 시스템이 특정 구간 때문에 느려지는 것

개발 과정에서 발생하는 작업들을 카드 형태로 만들고 진행 상황을 표시한다. 물리적 보드, 디지털 보드(JIRA, Trello, Redmine 등)을 사용해 운영.

3. 익스트림 프로그래밍 XP, Extreme Programming

![alt text](source/image.png)

개발자 중심의 방법론. 고객과 개발자가 함께 일하며 고객이 원하는 기능을 우선순위에 따라 작은 단위로 나누어 개발.

유연한 반복 개발 방식을 중심으로 하며 다음을 통해 고품질 소프트웨어를 만든다.

- 지속적 통합
- 단순한 설계
- 적극적인 리펙토링

- `페어 프로그래밍`: 두 명의 프로그래머가 함께 작업해 코드를 작성(1인 작성, 나머지 검토 및 피드백)
- `유닛 테스트+TDD`: 소프트웨어의 작은 부분인 유닛 단위로 테스트. (예상대로 작동하는지 확인)
- 페어 네고시에이션: 두 명의 사람이 함께 협상하여 문제를 해결하고 결정을 내리는 과정.
- `스탠드업 미팅`: 팀원들이 매일 짧은 시간 동안 서로의 진행 상황과 장애물을 공유하는 미팅.
- Acceptance Test: 소프트웨어가 요구사항을 만족하는지 확인하는 테스트
- Iteration Plan: 작은 주기로 소프트웨어를 개발하는 방법. 각 이터레이션마다 목표와 일정을 설정하고 이터레이션 동안 필요한 작업을 계획
- Release Plan: 제품 출시 계획. 기능 및 버그 수정 등의 작업 포함해 출시 일정과 전략 계획.

4. 크리스탈 패밀리: 팀 규모에 따라 프로세스를 조정하는 사람 중심 방법론.
5. FDD Feature-Driven Development: 기능을 중심으로 짧은 반복으로 개발하는 구조적 애자일 방법론.

## 장단점

### 장점

- 변화하는 요구사항에 민첩하게 대응 가능(개발 과정에서 지속적으로 요구사항을 재조정하므로)
- 고객 중심의 개발 철학 실현 가능
- 팀원들 간의 긴밀한 소통과 협업을 통해 창의적 아이디어와 해결책 도출 가능

### 단점

- 명확한 계획 수립이 어려움
- 요구사항이 빈번이 변경되며 개발 팀에 혼란을 줄 수 있음
- 대규모 프로젝트의 경우 애자일 프로세스 유지가 쉽지 않고 관리 복잡성 증가
