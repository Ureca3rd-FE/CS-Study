# 애자일 Agile

소프트웨어 개발 기법.

## 등장 배경

초기 소프트웨어 개발 방법은 계획 중심의 프로세스.(대표적 예시로 계획에 맞춰 차례차례 흘러가는 워터폴 방식) 하지만 트렌드가 급격하게 빨리 변화하게 되어 비지니스 사이클(제품 수명)이 짧아졌고 SW 개발의 불확실성이 높아짐.

덕분에 옛날의 전통적 개발 방법 적용이 어려워짐. -> 일단 해보고 고쳐나가는 방식 등장.

- 규칙은 적게
- 가볍게
- 바로바로 대응할 수 있도록

경량 방법론 주의자들이 노하우를 공유하고 공통점을 추려 애자일이란 방법론이 탄생.

## 그래서 애자일이란?

`반복적 개발`과 `점진적 개선`으로 피드백을 빠르게 반영해 제품을 지속적으로 발전시키는 접근 방식

### 하위 방법론

1. 스크럼 Scrum

시간 단위인 스프린트(2~4주)와 목표인 백로그 중심. 스프린트 동안 팀이 집중적으로 목표(백로그)를 달성하는 방법론.

2. 칸반 Kanban

시각적 도구를 활용해 업무 흐름을 한눈에 파악하게 하고 병목 현상을 조기에 발견, 해결

3. 익스트림 프로그래밍 XP, Extreme Programming

고품질 소프트웨어 개발을 위한 방법론으로

- 지속적 통합
- 페어 프로그래밍
- 테스트 주도 개발 TDD

등을 강조.

4. 익스트림 모델링?
5. 크리스털 패밀리?
6. FDD Feature-Driven Development?

### 가장 대표적인 방법: Scrum 진행 방법

전형적인 프로세스는 다음과 같음

1. 백로그 관리 Product Backlog Management

제품 요구사항을 수집하고 이를 우선순위에 따라 정리

2. 데일리 스크럼 Daily Scrum

매일 이루어지는 데일리 스크럼 회의로 진황 상황을 공유, 발생한 문제를 신속하게 논의

3. 스프린트 종료 후 스프린트 리뷰 Sprint Review

결과물을 시연하고 피드백을 수집

4. 회고 Retrospective

프로세스를 개선할 점을 도출

### 애자일 실천 방법

- 팀원 개개인이 자율성을 가지고 업무를 수행할 수 있어야 함(각자의 책임감 중요)
- 투명한 커뮤니케이션(모든 정보는 개방적으로 공유)
- 상명하달식 리더십보단 지원과 조율 중심의 리더십

## 장단점

### 장점

- 변화하는 요구사항에 민첩하게 대응 가능(개발 과정에서 지속적으로 요구사항을 재조정하므로)
- 고객 중심의 개발 철학 실현 가능
- 팀원들 간의 긴밀한 소통과 협업을 통해 창의적 아이디어와 해결책 도출 가능

### 단점

- 명확한 계획 수립이 어려움
- 요구사항이 빈번이 변경되며 개발 팀에 혼란을 줄 수 있음
- 대규모 프로젝트의 경우 애자일 프로세스 유지가 쉽지 않고 관리 복잡성 증가
