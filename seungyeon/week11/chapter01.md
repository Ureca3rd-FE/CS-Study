# 클린코드&리팩토링

## 클린코드
얼마나 가독성이 높은지, 지저분하지 않고 잘 정리된 코드인지를 의미.

클린 코드를 만드는 법
- 잘 된 네이밍
    - 다른 사람들이 봐도 무슨 역할을 하는 함수인지 알 수 있도록
    - 함수 사이에 간격을 두어 시작과 끝을 구분할 수 있도록
- 오류 없음
- 중복 없음
- 의존성 최대한 줄임
- 클래스 혹은 메소드가 한가지 일만 처리

## 리팩토링
외부 동작은 그대로 두고 내부의 코드를 정리하며 개선하는 것.

왜 하는가?
- 가독성 높이기
- 유지보수에 도움

성능 최적화를 위해 하는 거 X. 개발자가 읽고 이해하기 쉽게 만드는 것 O.<br>
일단 제대로 돌아가는 것이 중요, 리팩토링은 후순위다.

리팩토링이 필요한 곳
- 중복 코드
- 긴 메소드
- 거대한 클래스
- switch문
    - 객체 지향(OPP)스러움이 떨어짐. 왜?

        > 객체가 스스로 행동하지 못하고 외부에서 여러 타입을 분기해 같이 조작한다는 뜻이라 응집도가 떨어지고 결합도가 높아져 유지보수성이 떨어진다.
        
        조금 더 전문적으로 말하고 싶다면
        > `다형성`을 부정하고 `OCP(개방-폐쇄 원칙)`을 위반하기 쉽다.

        까지.
    - `다형성`: 다형성(Polymorphism)을 사용하면 객체에게 그냥 시키면 되는데 switch를 쓰면 객체 스스로 행동하지 않고 직접 행동을 지정하게 됨.

        > Animal이란 객체에 speak() 메소드를 통일하면 되는데 switch로 Dog인지 Cat인지 구분해서 bark()냐 meow()냐를 설정하는 등
    - `개방-폐쇄 원칙`: 객체 지향으로 설계하면 class 하나 상속으로 추가하면 끝인데 switch는 기능 추가에 따라 반드시 기존 코드를 수정해야함

        > 기능 추가는 열려있고, 수정은 닫혀있어야 함
    - switch는 한 곳에서 여러 타입의 행동을 모두 관리하니 응집도가 낮아지고 결합도가 높아짐
- 절차지향으로 구현한 코드

## 클린코드와 리팩토링의 차이
- 클린코드
    - 코드 작성 철학
    - 처음부터 잘 쓰기 원칙
    - 새 기능 포함 가능
    - 원칙, 규칙, 스타일
- 리팩토링
    - 코드 개선 활동
    - 기존 코드 품질 향상 과정
    - 기능/동작 변경 X
    - 구조, 설계 개선
