# TDD (Test Driven Development)

기존의 방법과 다르게 테스트 케이스를 먼저 작성한 후 실제 코드를 개발하는 개발 방식.

그렇게 반복적인 테스트와 수정을 통해 고품질의 소프트웨어가 탄생한다.

### 장점

- 작업과 동시에 테스트를 진행하면서 실시간으로 오류 파악 가능(시스템 결함 방지)
- 짧은 개발 주기를 통해 고객의 요구사항, 피드백 빠르게 수용 가능. 고객이 진행 사항 파악하기도 쉬움.
- 자동화 도구를 이용한 TDD 테스트 케이스를 단위 테스트로 사용이 가능 (JUnit, Jest 등)
- 개발자가 기대하는 앱의 동작에 관한 문서를 테스트가 제공
  - 또한, 테스트 케이스가 코드와 함께 업데이트 되므로 문서 작성이 어려워도 ok

### 단점

- 기존 개발 프로세스에 테스트 케이스 설계가 추가되므로 생산 비용 증가
- 테스트의 방향성, 프로젝트 성격에 따른 테스트 프레임워크 선택 등 추가로 고려할 부분이 많아짐

초기 비용이 많이 들고 귀찮지만 초기 비용보다 `유지보수 비용`이 더 많이 들 수도 있다. TDD 방식으로 개발하면 개발 초기 단계부터 확실하게 다져놓고 갈 수 있다.

(그래서 유지보수 비용이 많이 들거나 안전성이 중요한 프로젝트에서 자주 사용됨.)
