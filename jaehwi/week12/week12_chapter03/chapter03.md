# **써드 파티 (3rd party)**

## 써드 파티의 개념

<aside>
💡

- 다른 회사 제품에 이용되는 소프트웨어나 하드웨어를 개발하는 회사
    
    (프로그래밍을 도와주는 라이브러리를 만드는 외부 생산자)
    
- 우리 회사/팀이 직접 만들지 않고, **외부에서 제공**되는 모든 기술 요소
</aside>

서드파티는 **아래와 같은 외부 기술들을 모두 포함**하는 상위 개념

- 서드파티 라이브러리
- 서드파티 SDK
- 서드파티 API
- 서드파티 서비스 (클라우드 등)

### 써드 파티 구성 요소

> **서드 파티 라이브러리**
> 
- 개발 중 코드 재사용을 위해 포함하는 모듈이나 패키지
- ex) 프론트엔드의 React Router, 백엔드의 jsonwebtoken

> **서드 파티 SDK**
> 
- 외부 서비스를 애플리케이션에 쉽게 통합할 수 있도록 돕는 개발 키트
- ex) Firebase SDK, Facebook SDK

> **서드 파티 API**
> 
- 외부 서비스와 데이터를 주고받기 위한 인터페이스
- ex) 날씨 API, 결제 API, 소셜 로그인 API

> **서드 파티 툴/플랫폼**
> 
- 배포, 테스트, CI/CD 등 개발을 보조하는 외부 도구
- ex) Sentry(모니터링), Auth0(인증), Netlify(호스팅)

### **개발자 측면으로 본 써드 파티**

**퍼스트 파티 개발자**

 : 하드웨어 생산자가 '직접' 소프트웨어를 개발하는 경우

**세컨드 파티 개발자**

 : 하드웨어 생산자인 기업과 자사간의 관계(또는 하청업체)에 속한 소프트웨어 개발자

**써드 파티 개발자**

 : 아무 관련없는 제3자 소프트웨어 개발자

주로 편한 개발을 위해 `플러그인`이나 `라이브러리` 혹은 `프레임워크`를 사용…

 → 이처럼 제 3자로 중간다리 역할로 도움을 주는 것이 **써드 파티**, 이를 만드는 개발자가 **써드 파티 개발자**

## 써드 파티의 특징

> **기능 확장 가능**
> 

기본 프레임워크가 제공하지 않는 기능을 외부를 통하여 보완 가능

> **검증된 기능을 가져다가 사용**
> 

인증, 결제, 데이터 분석 복잡한 로직을 전문가가 만든 도구로 처리 가능

> **서비스 개발 속도 상승 (생산성 향상)**
> 

이미 검증된 기능을 재사용 → 개발 시간 단축

## 써드 파티 사용 기능 구분

### 써**드파티를 써야 하는 기능 (직접 구현 불가 or 위험)**

**보안·규제·기술 난이도 때문에 직접 구현하면 안 되는 영역**

- OAuth 소셜 로그인 (카카오/구글/애플)
- 지도 엔진 전체
- 결제 PG 연동
- OS 센서·카메라 기능
- 보안·암호화 프로토콜

### 써**드파티를 써도 되지만, 직접 만들어보면 좋은 기능**

직접 구현해보면 **기본기 성장 가능한 영역**

- 자체 로그인 (JWT)
- 파일 업로드 (S3 presigned URL)
- WebSocket 기반 채팅
- 푸시 알림 서버
- 필터/정렬/검색
- 비즈니스 로직 구현

### 써**드파티에만 의존하면 안 되는 핵심 영역**

**기업의 경쟁력이므로 개발자가 직접 만들어야 하는 영역**

- 서비스의 고유 기능
- 비즈니스 규칙
- 데이터 로직
- 주요 알고리즘
- 도메인 모델링
