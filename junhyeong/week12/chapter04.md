# Chap04. 마이크로서비스 아키텍처(MSA - Microservices Architecture)

## #0. 모놀리식 아키텍처

### 개념

- 모든 기능을 하나의 애플리케이션으로 구성하는 방식

### 모놀리식 아키텍처 vs 마이크로서비스 아키첵처

| 구분        | 모놀리식          | MSA              |
| :---------- | :---------------- | :--------------- |
| 구조        | 단일 어플리케이션 | 다수 서비스      |
| 배포        | 전체 재배포       | 서비스별 배포    |
| 확장        | 전체 확장         | 서비스별 확장    |
| 장애 영향   | 전체 서비스 영향  | 일부 서비스 영향 |
| 운영 난이도 | 낮음              | 높음             |

## #1. 마이크로서비스 아키첵처란?

### 개념

- 하나의 거대한 애플리케이션을 작고 독립적인 여러 서비스로 분리
- 각 서비스를 개별적으로 개발/배포/운영하는 아키텍처 방식

### 특징

#### 1. 서비스의 독립성

- 각 서비스는 자기 책임 범위(도메인)만 담당
- 서비스별로 개별 배포 가능
- 한 서비스 장애가 전체 시스템 장애로 이어지지 않음

#### 2. 느슨한 결합 (Loose Coupling)

- 서비스 간 통신은 HTTP/REST, gRPC, 메시지 큐 사용
- 내부 구현은 서로 알 필요 없음
- 기술 스택도 서비스마다 달라도 됨

#### 3. 데이터 분리

- 서비스마다 자신만의 DB 소유
- DB를 직접 공유하지 않음
- 데이터 일관성은 이벤트 기반으로 해결

#### 4. 확장성 (Scalability)

- 트래픽이 많은 서비스만 선택적으로 확장
- 서버 자원 효율 극대화

#### 5. 자동화·DevOps 친화적

- CI/CD 필수
- Docker, Kubernetes와 궁합이 좋음
- 배포/롤백이 빠름

## #2. 마이크로소프트 아키텍처의 예시

### 쇼핑몰 서비스

#### 서비스 및 역할

- 회원 서비스 <-> 로그인, 회원정보
- 상품 서비스 <-> 상품 조회, 재고
- 주문 서비스 <-> 주문 생성
- 결제 서비스 <-> 결제 처리
- 배송 서비스 <-> 배송 상태 관리

#### 기술 스택
- 회원 서비스: Spring Boot + MySQL
- 주문 서비스: Node.js + PostgreSQL
- 알림 서비스: Python + Redis
