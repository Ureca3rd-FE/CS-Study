# Chap05. 객체 지향 프로그래밍 (OOP - Object-Oriented Programming)

## #1. 객체지향 프로그래밍이란?

### 🔹개념

#### 1. 객체 (Object)

- 상태(State) + 행동(Behavior)을 가진 소프트웨어 단위
- 실제 세계의 개념을 프로그램에 대응시켜 모델링

#### 2. 클래스 (Class)

- 객체를 생성하기 위한 설계도
- 속성(필드)과 기능(메서드)을 포함한 추상화된 청사진

#### 3. 추상화 (Abstraction)

- 복잡한 시스템에서 핵심 개념만 추출하여 단순화
- 인터페이스/추상 클래스 등을 통해 "무엇을 할 수 있는지"를 정의하고 "어떻게"는 숨김

#### 4. 캡슐화 (Encapsulation)

- 데이터(필드)와 메서드를 하나로 묶고 외부에서 직접 접근하지 못하도록 보호
- 접근 제어자(public/private/protected)를 통해 정보 은닉 → 변경에 강한 구조

#### 5. 상속 (Inheritance)

- 부모 클래스의 속성과 기능을 자식 클래스가 물려받아 재사용
- 코드 중복 감소 + 확장 용이
- 오용할 경우, 강한 결합 발생 -> 조심해야하는 개념

### 🔹특징

#### 1. 높은 응집도 (High Cohesion)

- 하나의 클래스가 하나의 책임을 가지도록 설계
- 유지보수성 증가, 테스트 용이

#### 2. 낮은 결합도 (Low Coupling)

- 객체 간 의존 관계 최소화
- 변화가 발생해도 영향 범위가 적음

#### 3. 재사용성(Reusability) 증가

- 상속/구성(Composition)/인터페이스 등을 이용해 컴포넌트 재사용
- 대규모 시스템에서 개발 비용 절감

#### 4. 확장성(Extensibility) 강화

- 새로운 기능 추가 시 기존 코드 영향 최소
- 개방-폐쇄 원칙: 확장에는 열려 있고 수정에는 닫혀 있어야 함 (OCP)

#### 5. 유지보수성(Maintainability) 향상

- 시스템 이해가 쉬움 (실세계 모형과 유사)
- 모듈 간 경계가 명확해 디버깅·기능 변경이 수월

## #2. 객체지향 설계 원칙

### 🔹개념

#### 정의

- 객체지향 시스템을 품질 높게 설계하기 위해서는 따라야 하는 규칙이 있음
- 유지보수성, 확장성, 재사용성, 유연성을 극대화하는 것을 목표로 함
- 객체 간 역할과 책임을 적절히 분배하고, 변화에 강한 구조를 만들기 위해 설정한 설계 기준

#### 중요한 이유

- 유지보수 비용 감소
- 의도 파악 쉬움 -> 협업 용이
- 변경 시 영향 범위 최소
- 테스트 용이
- 재사용 가능한 구조 확보

### 🔹대표 원칙 - SOLID

#### S — SRP (Single Responsibility Principle)

- 단일 책임 원칙: 한 클래스는 하나의 책임만 가져야 한다.

#### O — OCP (Open–Closed Principle)

- 개방-폐쇄 원칙: 확장에는 열려 있고 변경에는 닫혀 있어야 한다.

#### L — LSP (Liskov Substitution Principle)

- 리스코프 치환 원칙: 자식 클래스는 부모를 대체할 수 있어야 한다.

#### I — ISP (Interface Segregation Principle)

- 인터페이스 분리 원칙: 필요한 기능만 제공하도록 인터페이스를 분리한다.

#### D — DIP (Dependency Inversion Principle)

의존 역전 원칙: 구체가 아닌 추상(인터페이스)에 의존해야 한다.
