# 11주차 🔧소프트웨어 엔지니어링

# Chap04. 애자일 (Agile)

## #0. 등장 배경

### 🔹소프트웨어 개발 초창기 - 1950~1960년대

#### 체계적인 절차가 없었음

- 문서, 계획의 개념이 명확하지 않았음
- 개발자 개인에게 의존함

#### 체계적인 프로세스의 필요성이 대두됨

- 절차 부재의 결과, 문제 발생 시 해결이 어려움
- 프로젝트 규모가 커질수록 관리가 힘들어짐
- 소프트웨어 개발도 공학적으로 관리해야 한다는 인식이 등장함

### 🔹폭포수 모델(Waterfall Model)의 등장 - 1970년대

#### 규칙적인 공정

- Winston Royce가 논문에서 폭포수 개념을 소개
- 소프트웨어 개발에 처음으로 체계적이고 단계적인 절차가 도입됨
- 각 단계가 폭포수처럼 위에서 아래로 흐르는 구조

#### 폭포수 모델의 핵심

1. 요구사항 분석
2. 설계
3. 구현
4. 테스트
5. 배포
6. 유지보수

#### 폭포수 모델의 장점

- 문서화가 정확함
- 책임 구분이 쉬움
- 대규모 프로젝트에서 통제가 가능함
- 제조업 방식과 유사 -> 이해하기 쉬움

#### 폭포수 모델의 한계

- 소프트웨어는 제조업과 달리 변화가 매우 많은 분야
- 요구사항이 변경 시 앞 단계로 되돌아가기 어려움: 변경 비용 증가
- 고객은 개발이 끝나기 전까지 결과물을 볼 수 없음
- 거대한 문서 중심 문화: 제품보다 문서가 먼저 -> 개발 속도 저하
- 시장 변화에 적응 불가: 인터넷, UI/UX가 빠르게 변하는 시대에 민첩한 대응이 어려움

### 🔹애자일의 탄생 - 2001년

#### 1990년대 후반부터 새로운 방법론이 등장

- XP(eXtreme Programming)
- 스크럼 (Scrum)
- Crystal
- Adaptive Software Development

#### 새로운 방법론들의 특징

- 개발 주기가 짧음
- 빠른 피드백
- 문서보다 소통을 중시
- 변화에 대응하는 것을 우선시함

#### 2001년 애자일의 탄생

- 위 흐름들이 모여 2001년 17명의 소프트웨어 개발자들이 모여 애자일 선언문을 작성하고 발표함
- 애자일의 핵심 가치 4가지
  1. **개인과 상호작용 > 프로세스와 도구**
  - 정해진 절차/도구보다 팀원 간의 소통과 협력이 중요
  2. **작동하는 소프트웨어 > 포괄적인 문서**
  - 문서 작성에 시간을 쓰느라 실제 개발 속도가 늦어져서는 안됨
  3. **고객과의 협업 > 계약 협상**
  - 개발 중에도 고객과 계속 소통하고 문제를 해결하는 것이 더 중요
  4. **변화 대응 > 계획 준수**
  - 바뀐 요구에 빠르게 대응하는 것이 계획을 지키는 것보다 중요

## #1. 애자일(Agile)이란?

### 🔹개념

- 빠르게 변화하는 요구사항에 유연하게 대응하기 위해 만들어진 소프트웨어 개발 방식
- 짧은 반복 주기(스프린트)로 조금씩 개발하고, 즉시 피드백을 받아 다음 작업에 반영하는 방식

#### 애자일의 목표

- 빠르게 작동하는 소프트웨어 제공
- 그 과정에서 고객/팀원과의 지속적인 소통
- 변화된 요구를 자연스럽게 받아들이는 것

### 🔹협력 (Cooperation)

#### 의미

- 애자일의 핵심 가치 중 하나인 **개인과 상호작용 > 프로세스와 도구**를 강조
- 사람 간의 소통이 개발 성공의 핵심이라는 개념

#### 필요성

1. **요구사항이 개발 도중에 바뀜**

   - 문서만 보고 개발하면 변화 대응에 느림
   - 직접 대화하면 빠르게 이해 및 해결 가능

2. **역할 간의 긴밀한 협업이 필요**

   - 개발자/디자이너/기획자/고객이 자주 만나 아래의 요소를 즉각적으로 확인
     - 무엇을 만들지
     - 어떻게 만들지
     - 지금 방향이 맞는지

3. **문제 상황을 빠르게 해결**
   - 이슈가 생기면 즉시 팀이 모여 해결책을 논의
   - 대기 시간 감소 -> 개발 속도 향상

### 🔹피드백 (Feedback)

#### 의미

- 애자일은 보통 1-2주 단위의 짧은 개발 주기(스프린트)를 사용
- 각 주기 끝에 작동하는 결과물을 사용자/고객에게 직접 보여주고 즉각적인 피드백을 받음

#### 필요성

1. **고객의 진짜 요구를 빠르게 파악 가능**
   - 처음 문서에 적힌 요구가 고객이 원하는 것과 다를 때가 많음
   - 동작하는 결과물을 보고 진짜 요구사항 파악 가능 -> 다를 경우 즉시 수정
2. **잘못된 방향으로 오랫동안 개발하는 위험 줄임**
   - 작은 단위로 만들고 바로 검증
   - 잘못된 경우 바로 방향을 수정
3. **팀 내부 품질 향상**
   - 코드 리뷰
   - 데일리 스크럼에서 진행 상황 공유
   - 회고로 더 나은 방식 찾기
4. **빠른 실패**
   - 빨리 실패하고 빨리 고치기
   - 전체적인 성공 확률을 높이는데 기여

## #2. 애자일의 진행 방법

#### 진행 방식

- 애자일은 짧은 주기(스프린트)로 개발하고 그때그때 피드백을 반영하며 개선해 나가는 방식
- 스크럼 방법론 기반으로 설명하는 것이 가장 표준적인 절차

#### 애자일 진행 방법의 핵심

1. 짧은 주기 (스프린트)
2. 작동하는 결과물 중심
3. 지속적인 피드백 반영
4. 팀의 자율성과 협력 강조

### 애자일 개발 사이클 (Agile Development Cycle)

#### 1. 제품 백로그 작성 (Product Backlog)
- 제품에 필요한 모든 기능/요구사항 목록
- e.g. 로그인 기능, 장바구니 기능, UI 개선
- 문서보다는 해야할 일 리스트의 느낌
- 우선순위는 PO(Product Owner)가 관리

#### 2. 스프린트 계획 회의 (Sprint Planning)
- 보통 1-2주 단위로 스프린트 시작 시 진행
- 팀이 이번 스프린트에서 처리할 작업(백로그) 선택
- 스프린트 목표 설정
- 각 작업은 태스크로 쪼개어 예측 가능하고 수행 가능한 범위로 만듦

#### 3. 스프린트 진행 (Sprint)
- 실제 개발/디자인/테스트가 진행되는 기간
- 스프린트 동안은 목표를 바꾸지 않음
- 팀은 매일 짧은 회의(데일리 스크럼)를 통해 진행 상태를 점검
    1. 어제 한 일
    2. 오늘 할 일
    3. 장애 요소 (Blocker)

#### 4. 스프린트 리뷰 (Sprint Review)
- 스프린트가 끝나고 고객/PO에게 실제 동작하는 결과물을 시연
- 피드백을 받고 바로 다음 스프린트 계획에 반영

#### 5. 스프린트 회고 (Sprint Retrospective)
- 팀 내부 개선을 위한 시간
- 논의 내용: 잘된 점 / 아쉬웠던 점 / 다음 스프린트에 개선할 점

#### 6. 다음 스프린트 시작
- 새 스프린트를 시작하며 반복
- 반복적/점진적 개발이 핵심
- 매 스프린트마다 제품 품질 및 팀 프로세스가 함께 성장
