# 이상(Anomaly)

정규화를 하는 이유는 잘못된 테이블 설계로 인해 **이상 현상(Anomaly)** 이 발생하기 때문이다.  
이상 현상은 데이터의 **삽입, 갱신, 삭제 과정에서 비정상적인 동작이나 데이터 불일치가 일어나는 현상**을 말한다.

> **예시 테이블**  
> {Student ID, Course ID, Department, Course Name, Grade}
>
> **기본키**  
> {Student ID, Course ID}

## 삽입 이상 (Insertion Anomaly)

**설명**  
학생이 아직 수강하지 않은 상태라면 `Course ID`가 없음 → 기본키 일부가 `NULL`이 되어 튜플을 삽입할 수 없음.

**예시**  
Course를 수강하지 않은 학생 정보를 넣고 싶을 때 `Course ID`가 비어있으면 삽입 불가함.  
억지로 "미수강" 같은 임의 값을 넣어야 함.

**결과**  
불필요한 데이터를 추가해야 삽입이 가능한 상황 발생 → 삽입 이상이 발생함.

## 갱신 이상 (Update Anomaly)

**설명**  
학생의 학과가 변경될 경우 한 학생의 정보가 여러 튜플에 중복되어 있다면 **모든 튜플을 수정해야 함.**

**예시**  
Student A의 학과가 “컴퓨터공학” → “음악학”으로 바뀌면 모든 튜플의 학과 값을 수정해야 함.  
하나라도 빼먹으면 데이터 불일치가 발생함.

**결과**  
일부 데이터만 수정되어 데이터 불일치 발생 → 갱신 이상이 발생함.

## 삭제 이상 (Deletion Anomaly)

**설명**  
학생이 수강을 철회하면 해당 수강 정보뿐 아니라 학생 정보까지 함께 삭제되는 문제임.

**예시**  
수강 철회 시 튜플 전체가 삭제되면서  
`Student ID`, `Department` 등 학생의 고유 정보까지 사라짐.

**결과**  
필요한 데이터까지 함께 삭제되는 문제 → 삭제 이상이 발생함.
